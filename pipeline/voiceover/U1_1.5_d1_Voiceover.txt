[Slide 6]
Three uses, and notice they line up with the objective — networks, applications, data. First: AI reviews security configurations, the firewall rules and access controls that protect a network, and recommends tighter, more secure settings. But a knowledgeable security technician checks those recommendations before they go live. Second: AI analyzes application code, finds vulnerabilities, and recommends fixes — and a knowledgeable programmer reviews them before they ship. Third: AI suggests rules for automated detection systems, and a knowledgeable detection engineer reviews each rule before it is added. Hear the pattern? Review, recommend, and a human signs off — every single time. Write all three, and write the human who approves each.

[Slide 7]
Here is Scenario 1E end to end, and it is the cleanest illustration you'll get. Step one, the find: the company is launching a web app that takes customer orders and updates a warehouse database. Before launch, they ask an AI tool to review the code, and it flags several places where a customer's typed input is sent straight into database requests — exactly the kind of hole an adversary could use to read the inventory or modify the database. Step two, the human approval: the AI recommends fixes that validate and sanitize that input, and those recommendations go to the software development team, who review the changes and update the code appropriately. Then, and only then, the app moves to testing and later to deployment. On the exam, name both halves — AI found the vulnerability and recommended the fix, and a knowledgeable programmer reviewed it before it shipped. The review is not optional.

[Slide 8]
This is the misconception the College Board built directly into the standard, which is why it tests it. The AI hands you a finished-looking fix — it names the bug, explains the danger, gives you the code — so the instinct is to paste it and move on, especially when you're racing to launch. But read the standard: every one of the three says the recommendation should always be checked by a knowledgeable human before it's implemented. Why? Because the AI might patch one field and miss the next, or break a feature, or misunderstand how your system really works. In Scenario 1E the dev team reviews the changes for exactly that reason. The frame to carry out of here: on defense, AI is the advisor that makes a human expert faster — it is never the expert that replaces them. That distinction is worth points.

[Slide 9]
Quick enrichment, because it ties the whole unit together. On the left is yesterday: adversaries using AI to scan large code bases for vulnerabilities to exploit, and to write malware. On the right is today: defenders using that identical capability to scan their own code for the same vulnerabilities — to fix them before launch, exactly like Scenario 1E. The tool doesn't care which side runs it; code analysis surfaces the same bug either way. Two things differ. One is intent — exploit versus patch. The other is that the defender adds a human review step the attacker never bothers with. So the practical lesson is a race: whoever scans first and closes the hole first wins. Defenders who use AI well get to find their bugs before the adversary's AI does.

Know all six by name.
